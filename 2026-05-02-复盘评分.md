## 四、核心问题分析

### 🔴 问题1：数据获取形同虚设，报告"带病上岗"（致命）

**现象：**
- 报告"大盘概览"标注"数据待获取"，但仍输出了完整推荐内容
- 报告在09:00生成时数据获取失败，脚本 fallback 到昨日收盘价，伪装成"今日数据"
- 仓位建议仍然输出"50-70%"，完全无视 stock.md v2.9 的硬约束规则

**违反规则：**
- stock.md 第零部分 v2.9：**"若任一指数获取失败 → 输出'今日数据不可用，请勿执行任何操作'，自动将版本标记为'未验证'"**
- 实际行为：输出了完整报告 + 乐观情绪 + 50-70%仓位

**根因：** generate_report.py 没有实现数据获取失败时的 assert 硬终止逻辑，stock.md 规则是"软约束"而非"硬约束"

---

**现象：**
- 报告头部：`AAna v2.3`

**影响：**

## 五、stock.md v3.0 优化建议

### 🔧 优化1：数据前置检查 → 硬约束（最高优先级）

**问题：** 虽然规则写了"数据不完整则拒绝生成报告"，但脚本没有实现强制终止逻辑。

**v3.0 方案：**
```python
# 在 generate_report.py 中实现
assert all_indices_success, "大盘指数数据获取失败，禁止生成报告"
assert market_data['timestamp'] is not None, "数据时间戳缺失"
```

**修改 stock.md：**
- 将"数据完整性"规则从"软约束（标注警告）"升级为"硬约束（脚本 assert）"
- 数据缺失时必须输出以下内容并终止报告生成：
  ```
  ⚠️ 今日数据不可用，请勿执行任何操作
  数据状态：上证指数❌ / 深证成指❌ / 创业板指❌ / 科创50❌
  仓位建议：0%（禁止开仓）
  ```
- 即使是 cron 17:00 的自动复盘运行，也必须先确认数据来源（不要 fallback 到昨日数据）

---

### 🔧 优化2：版本一致性自动校验（最高优先级）

**问题：** 报告版本 v2.3 ≠ stock.md 版本 v2.9

**v3.0 方案：**
```python
# 在 generate_report.py 中实现
with open(f"{PROJECT_DIR}/docs/stock.md", "r") as f:
    stock_md = f.read()
import re
version_match = re.search(r'version[:\s]+v?(\d+\.\d+)', stock_md, re.IGNORECASE)
script_version = version_match.group(1) if version_match else "unknown"
report_version = get_version_from_script()  # 读取脚本内的版本号
assert script_version == report_version, f"版本不一致: script={script_version}, report={report_version}"
```

**修改 stock.md：**
- 增加明确的版本号头部格式：`> 版本：v3.0（复盘优化：...)`
- 脚本读取版本时支持正则 `version[:\s]+v?(\d+\.\d+)`

---

### 🔧 优化3：追高过滤 → 强制执行 + 自动降权

**问题：** 27只推荐股全部违反追高规则，但只提示风险、未执行过滤。

**v3.0 方案：**
```python
# 在 generate_report.py 中实现
def apply_chase_filter(candidates):
    filtered = []
    for stock in candidates:
        today_pct = stock['change_pct']
        near5_pct = stock.get('near5_change_pct', 0)
        is_high = stock.get('price_near_20d_high', False)
        
        # v2.9规则：自动移除
        if today_pct > 7 and near5_pct > 10 and is_high:
            continue  # 完全移除，不只是降权
        
        # v2.9规则：降权到末尾
        if today_pct > 5 and near5_pct > 7:
            stock['priority'] = 'low'  # 降权
            filtered.append(stock)
        else:
            filtered.append(stock)
    return filtered
```

**修改 stock.md：**
- 明确"追高过滤"是**强制过滤**，不是可选提示
- "风险提示"章节的追高股票必须与"推荐列表"完全互斥——出现在风险提示就不应该出现在推荐列表

---

### 🔧 优化4：动态选股池 vs 固定关注列表

**问题：** fetch_stock_data.py 只抓取14只预设股票，无法发现新强势股。

**v3.0 方案：**
- generate_report.py 增加**全市场扫描逻辑**（调用 eastmoney_scraper.py 的板块涨幅排行）
- 优先从**近5日涨幅前3板块**中各取前5只个股
- 避免只依赖昨日收盘价的固定股票池

**修改 stock.md：**
- 选股池必须来自：①板块动量扫描 ②龙头股强制纳入 ③用户自定义关注
- 推荐列表容量限制：Top10 来自回调股（涨幅<3%），最多允许3只追高股但必须标注"仅观察"

---

### 🔧 优化5：仓位建议必须基于实际数据

**问题：** 数据缺失时仍给出"50-70%乐观仓位"。

**v3.0 方案：**
- 数据缺失 → 强制仓位 0%，输出"数据不可用，禁止开仓"
- 数据完整但科创50/创业板跌幅>1.5% → 强制进入警报模式，仓位≤30%
- 永远不允许在数据未校验的情况下给出"积极模式/50-70%"建议
